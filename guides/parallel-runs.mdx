A single TuLIP sill cube contains many 2D cross-sectional slices — each one a valid, independent simulation. The natural way to exploit this is to assign each slice to a separate CPU core and run them all simultaneously. TuLIP's `utilities` module ships two drop-in functions, `cooler()` and `sillburp_cooler()`, designed specifically to be used with joblib's `Parallel`/`delayed` pattern. Each function loads a pre-built sill cube slice, runs the full emplacement-plus-cooling physics, and writes its own output directory — no shared state between workers.

## How it works

`utilities.cooler()` is a self-contained simulation runner. When you call it, it:

1. Reads the initial properties array and maturation state from pre-saved VTK files in `file_path_dir`.
2. Loads the correct 2D sill slice for the given `flux`, `iter`, and `z_index`.
3. Calls `sc.emplace_sills()` with the SILLi carbon model.
4. Writes a `times.csv` containing `time_steps`, `tot_RCO2`, melt fractions, and sill area to a per-run output directory.

`utilities.sillburp_cooler()` is the counterpart for the Sillburp multi-kinetics emissions model. It reads `initial_sillburp_state_*.vtk` files and loads additional reaction-progress arrays (`progress_of_reactions.npy`, `rate_of_reactions.npy`).

## `utilities.cooler()` parameters

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `iter` | `int` | — | Row index into `n_sills.csv`; selects the total-volume scenario. |
| `z_index` | `int` | — | Along-strike slice index through the 3D sill cube. |
| `flux` | `float` | — | Magma supply rate (m³/yr); used to build the load directory path. |
| `lat_range` | `list | None` | `None` | Lateral range filter used when building the load path. |
| `sc` | `sill_controls` | `None` | Shared `sill_controls` instance. Required. |
| `diff_val` | `float` | `31.536` | Reference thermal diffusivity (m²/yr) used only for fallback dt calculation. |
| `temp_grad_base` | `float` | `0.03` | Basal geothermal gradient (°C/m) for computing the bottom heat flux `q`. |
| `file_path_dir` | `str` | `'sillcubes/'` | Root directory containing sill cubes and initial-state VTK files. |
| `post_cooling_time` | `float` | `30000` | Years of additional cooling modelled after the last sill emplacement. |
| `saving_factor` | `list[int]` | `[100]` | VTK snapshot frequency: write every `saving_factor[0]` time steps. Pass two values `[N_during, N_after]` to use different rates during and after emplacement. |
| `x`, `y`, `dx`, `dy` | `float` | `None` | Domain dimensions; must match the `sill_controls` object. |
| `rock_prop_dict` | `dict | None` | `None` | Override rock property dictionary. Falls back to `sc.rock_prop_dict`. |
| `save_dir_footnote` | `str | None` | `None` | String appended to the output directory name to distinguish experiment variants. |

## `utilities.sillburp_cooler()`

Use `sillburp_cooler()` instead of `cooler()` when your initial maturation state was computed with `get_sillburp_initial_thermogenic_state()`. It reads `initial_sillburp_state_properties.vtk`, `initial_sillburp_state_carbon.vtk`, `progress_of_reactions.npy`, `rate_of_reactions.npy`, and optionally `sillburp_weights.csv` from `file_path_dir`. The call signature is identical to `cooler()`, and it passes `model='sillburp'` to `emplace_sills()` internally.

```python sillburp_example.py
Parallel(n_jobs=30)(
    delayed(util.sillburp_cooler)(
        iter_1, z_index_1, fluxy_1,
        sc=sc, x=x, y=y, dx=dx, dy=dy,
        file_path_dir=file_path_dir,
        post_cooling_time=post_cooling_time,
        saving_factor=saving_factor,
    )
    for iter_1, z_index_1, fluxy_1 in pairs
)
```

<Tip>
  Build your sill cubes and generate the 2D slice files **before** launching the parallel run. Call `sc.generate_sill_2D_slices(fluxy_list, iter_list, z_index_list, file_dir=file_path_dir)` once to extract and save each cross-section as a separate `.npy` file. Workers load these slices at startup — no cube rebuilding occurs during the parallel run.
</Tip>

<Warning>
  Each worker loads its own copy of `props_array`, the sill cube slice, and all carbon-model state arrays into RAM. With 30 workers and a 300 × 12 km domain at 50 m spacing (6000 × 240 grid), peak memory per worker is roughly 500 MB. Multiply by `n_jobs` to estimate total RAM demand before choosing a core count. If you exceed available memory, joblib will silently slow down due to swap usage; reduce `n_jobs` accordingly.
</Warning>
